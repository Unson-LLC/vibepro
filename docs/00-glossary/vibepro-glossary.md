# VibePro 用語集

## 本書の位置づけ

この文書は、Brainbase Graph SSOT に登録された VibePro の思想、DAG、指標用語を repo 内で参照するための投影である。

正本は Brainbase Graph SSOT の `glossary_term` である。主な登録IDは次の形式を使う。

- `vibepro.term.*`
- `vibepro.metric.*`

VibePro では、AI が自分で判断し、後から自己採点し、次の判断へフィードバックできることを前提に用語を定義する。曖昧な略語や英語名を正本にしない。

---

## 1. 判断構造

### 発火理由

- 定義: なぜ今その作業を開始するのかを示す理由。
- 例: 新規立ち上げ、障害予兆、精度低下、コスト増、監査要求、顧客要望。
- ルール: 発火理由がない作業は進めない。

### 状態

- 定義: 現在の対象プロダクト、顧客、運用、リスク、情報品質の状況。
- 例: 要件明確度、プロダクト段階、AI依存度、データリスク、連携複雑度。
- ルール: 状態が不明な作業は実装に流さない。

### 判断

- 定義: 発火理由と状態の組み合わせから、次に流す経路を決めること。
- 例: 診断、Story 化、Architecture 設計、Spec 化、実装、人間ゲート、除外。
- ルール: 判断は実装層で後付けしない。

### 状態クラスタ

- 定義: 対象を同じ判断ルールで扱える状態のまとまりに分類したもの。
- 例: `beta / AI依存度 high / 運用リスク medium / データリスク sensitive`。
- 用途: 低リスクは自動化へ流し、高リスクは人間ゲートへ流す。
- ルール: 状態クラスタを切る前に、テンプレートや実装手段を決めない。

### 品質状態

- 定義: 文脈、Story、Architecture、Spec、テスト可能性、証跡の品質を表す状態。
- 用途: 警告ではなく、ルーティング条件として使う。
- 例: 文脈品質が低い場合は診断へ戻す。Spec品質が低い場合は実装に入らない。

## 2. DAG とゲート

### 文脈 DAG

- 定義: 入力情報、事実、制約、SSOT を集める層。
- 禁止: 意味判断を確定すること。

### Story DAG

- 定義: 誰が、何を、なぜ達成したいかを固定する層。
- 禁止: API、DB、実装詳細を書くこと。

### Architecture DAG

- 定義: 境界、責務、正本、制御面を決める層。
- 禁止: コードや schema を確定すること。

### Spec DAG

- 定義: event、command、state、API、schema、テスト条件へ落とす層。
- 禁止: Story を後付けで変更すること。

### 実行 DAG

- 定義: TDD、実装、検証、リリース準備を行う層。
- 禁止: 未承認の判断を実装で埋めること。

### 証跡 DAG

- 定義: 実行結果、テスト、ログ、リスク、差分を残す層。
- 禁止: 成功/失敗の根拠を曖昧にすること。

### 学習 DAG

- 定義: 運用 KPI と改善 Story に接続する層。
- 禁止: 単発納品で終わらせること。

### 昇格ゲート

- 定義: 作業状態を次の段階へ進めてよいかを判定する条件。
- 例: `Signal -> Intake -> Diagnosed -> Story Ready -> Architecture Ready -> Spec Ready -> Execution Ready -> Shipped -> Verified -> 学習済み`。
- ルール: 昇格条件を満たすまで次の段階へ進めない。

### 人間ゲート

- 定義: AI に任せてよい範囲を事前に決める制御点。
- 必須例: 本番データ移行、本番切替、顧客通知、契約責任が変わる判断、セキュリティ例外、ロールバック不能な変更。
- ルール: 人間ゲートが必要な判断を、実装差分で既成事実化しない。

## 3. AI自走指標

### AI自走指標

- 定義: AI が自分で判断し、後から自己採点し、次の判断へフィードバックできる指標。
- 条件:
  1. 入力が機械的に取得できる
  2. 出力が構造化されている
  3. 後から正解ラベルまたは検証結果を取得できる
  4. 差分を次の判断ルールへ戻せる

### 正解ラベル

- 定義: AI の事前判断が正しかったかを後から照合するための構造化された結果。
- 例: 実装中に発見された追加ギャップ、リリースを止めたリスク、運用後に発生したインシデント、効果が確認された改善介入。
- ルール: 正解ラベルが取れない指標は、AI の自己改善指標として扱わない。

### 本番化ギャップ

- 定義: AIプロダクトを本番で安全に運用するために不足している設計、実装、運用、セキュリティ、監視、評価、コスト管理上の差分。
- 例: RBAC不備、監視なし、ロールバック手順なし、評価セットなし、API変更追従なし、コスト上限なし。

### 本番化ギャップ捕捉率

- 定義: 実装・運用で実際に重要だった本番化ギャップのうち、Launch診断時点で検出できていた割合。
- 計算: `診断時点で検出できた重要ギャップ数 / 実際に重要だったギャップ数`。
- 目的: 見落としを減らす。
- 注意: この指標だけを最大化すると過剰警告が増えるため、本番化ギャップ的中率とセットで見る。

### 本番化ギャップ的中率

- 定義: Launch診断で挙げた本番化ギャップのうち、実装・運用で実際に重要だった割合。
- 計算: `実際に重要だった診断ギャップ数 / 診断で挙げたギャップ数`。
- 目的: 過剰警告を減らす。
- 注意: この指標だけを最大化すると見落としが増えるため、本番化ギャップ捕捉率とセットで見る。

### ゲート違反流出率

- 定義: 本来 Gate で止めるべきだった変更・判断のうち、実装・リリース・運用へ流出した割合。
- 計算: `流出したゲート違反数 / 本来止めるべきだったゲート違反数`。
- 目的: 危険な自動化を止める。
- 評価: 低いほどよい。

### ルート成果合致率

- 定義: 診断時に推奨したルートと、実際に成功したルートが一致した割合。
- 例: `Launch実装 S` と推奨し、実際にも S の範囲で完了した場合は合致。
- 用途: ルーティング判断の自己採点。

### 受け入れ条件網羅率

- 定義: Story の受け入れ条件が、Spec、Test、Evidence に接続された割合。
- 用途: Story から実装までの抜け漏れ検出。

### 仕様逸脱率

- 定義: 実装結果が Spec から逸脱した割合。
- 用途: 実行 DAG の品質評価。
- 評価: 低いほどよい。

### 前進修正率

- 定義: rollback ではなく、小さな追加修正で前進できた割合。
- 用途: 実装と運用の安定性評価。

### 手戻り率

- 定義: 実装開始後に Story、Architecture、Spec へ戻った割合。
- 用途: 前段判断の粗さを測る。
- 注意: 手戻り自体を悪としない。重要なのは、手戻り理由が構造化され、次回の判断へ戻ること。

### 検証証跡合格率

- 定義: 必須証跡が揃い、再現可能で、監査可能だった割合。
- 用途: Shipped / Verified 判定の品質評価。

### 介入別KPI改善率

- 定義: 各改善介入が、品質、コスト、体験、信頼性、変更耐性の KPI をどれだけ改善したか。
- 用途: Growth Ops の自己改善。

## 4. リスク指標

### 最大手戻り

- 定義: 1案件または1改善単位で発生した最大の戻り工程。
- 例: Execution Ready から Story Ready まで戻る。

### 最大納期遅延

- 定義: 予定納期に対して発生した最大遅延。

### 最大障害影響

- 定義: 障害が顧客、ユーザー、売上、信頼性に与えた最大影響。

### 最大コストスパイク

- 定義: 通常水準から最も大きく跳ねたコスト増加。

### 最大未検出リスク

- 定義: 診断時に検出できず、後から重要問題として顕在化した最大リスク。

---

## 5. 禁止する混同

- 本番化ギャップ捕捉率と本番化ギャップ的中率を混同しない
- ルート成果合致率と事業成果を混同しない
- 検証証跡合格率と品質改善を混同しない
- 手戻り率と失敗を混同しない
- Growth KPI と DAG の判断品質を混同しない
- 人間向けレポート指標を AI自走指標として扱わない
