# VibePro 運用思想

## 本書の位置づけ

この文書は、Brainbase Graph SSOT の `frame: frm_vibepro` を VibePro repo に投影した参照文書である。

VibePro は、AI プロダクトを速く作るだけのサービスではない。AI プロダクトを、事業で使える状態、本番で止まらない状態、継続的に改善できる状態へ進めるための control plane である。

用語の正本は Brainbase Graph SSOT の `glossary_term` とする。[VibePro 用語集](../00-glossary/vibepro-glossary.md) は repo 内参照用の投影である。

VibePro の中心価値は、実装量ではなく、次の 3 つを安定して回すことにある。

1. 何を進めるべきかを判断する
2. どの条件なら AI と自動化に任せてよいかを制御する
3. 完了後に証跡を残し、次の改善へ接続する

VibePro が一番達成したいことは、AI に開発を丸投げすることではない。

Story から Architecture が導かれ、その Architecture を人間が確定できたら、その後の実装・検証・修正を AI に安心して任せられる状態を作ることである。

AI に任せたいのは、実装作業、検証作業、細かい修正作業である。AI に任せないものは、プロダクトの意味、責任境界、構造判断、完成の定義である。

VibePro が守る主権は 2 つある。

1. 設計の主権
   - 何を作るべきか、なぜ作るべきか、どの責任境界で作るべきかを人間が確定する。
   - AI は確定した Story / Architecture / Spec の中で実装する。
   - AI が構造判断を実装差分で既成事実化することを許さない。
2. 完成度の主権
   - 完成したかどうかを AI の自己申告やコード差分だけで決めない。
   - ユーザーが UI を触り、意図した操作・遷移・保存・理解に到達できる品質を完了条件に含める。
   - 最後の 20% を人間が毎回手作業で埋める状態を、VibePro の未解決課題として扱う。

VibePro は、AI が速く 80% を作ることよりも、その後に 95% の実利用品質まで仕上げ切れることを重視する。

この 95% 品質とは、テストが通ることだけではない。ボタンが押せる、画面が遷移する、入力が保存される、リロードしても破綻しない、モバイル幅で崩れない、エラーや空状態で迷わない、ユーザーが人間として触って理解できる、という E2E レベルの体験品質である。

## 1. 発火理由 x 状態 -> 判断

VibePro は、曖昧な要望をそのまま実装に流さない。

すべての作業は、次の 3 点を分けて扱う。

| 要素 | 意味 | 例 |
|------|------|-----|
| 発火理由 | なぜ今動くのか | 新規立ち上げ、障害予兆、精度低下、コスト増、監査要求、顧客要望 |
| 状態 | 今どんな状態か | 要件明確度、プロダクト成熟度、リスク、依存関係、データ品質、運用成熟度 |
| 判断 | どの経路に流すか | 診断、Story 化、Architecture 設計、Spec 化、実装、人間ゲート、除外 |

原則:

- 発火理由がない作業は進めない
- 状態が不明な作業は実装に流さない
- 判断は発火理由と状態の組み合わせで決める
- 「やりたいこと」だけでは Story Ready とみなさない

## 2. レイヤー分離された制御 DAG

VibePro の実行は、単一の巨大な自動化ではなく、責務を分けた DAG として扱う。

```text
文脈 DAG
  -> Story DAG
  -> Architecture DAG
  -> Spec DAG
  -> 実行 DAG
  -> 証跡 DAG
  -> 学習 DAG
```

| Layer | 責務 | 禁止すること |
|-------|------|--------------|
| 文脈 DAG | 入力情報、事実、制約、SSOT を集める | 意味判断を確定する |
| Story DAG | 誰が、何を、なぜ達成したいかを固定する | API、DB、実装詳細を書く |
| Architecture DAG | 境界、責務、正本、制御面を決める | コードや schema を確定する |
| Spec DAG | event、command、state、API、schema、テスト条件へ落とす | Story を後付けで変更する |
| 実行 DAG | TDD、実装、検証、リリース準備を行う | 未承認の判断を実装で埋める |
| 証跡 DAG | 実行結果、テスト、ログ、リスク、差分を残す | 成功/失敗の根拠を曖昧にする |
| 学習 DAG | 運用 KPI と改善 Story に接続する | 単発納品で終わらせる |

VibePro の成熟度は、AI の賢さではなく、DAG がどれだけ安全に広い範囲を扱えるかで測る。

## 3. 状態クラスタを先に切る

改善や実装方針を決める前に、対象を状態クラスタに分ける。

VibePro で最低限見る状態クラスタ:

| 状態クラスタ | 観点 |
|--------------|------|
| プロダクト段階 | idea / prototype / beta / production / growth |
| 要件明確度 | unclear / partial / clear |
| AI依存度 | low / medium / high |
| 運用リスク | low / medium / high |
| データリスク | none / light / sensitive / regulated |
| 連携複雑度 | standalone / few integrations / many integrations |
| 移行リスク | none / reversible / high blast radius |
| 人間ゲート要否 | none / approval / mandatory |

原則:

- 状態クラスタを切る前に、テンプレートや実装手段を決めない
- 低リスククラスタは自動化へ流す
- 高リスククラスタは人間ゲートと別スコープ審査へ流す
- 状態クラスタが変わったら、Story / Architecture / Spec のどこへ戻すかを明示する

## 4. 品質状態はルーティング条件

品質状態は、警告やメタ情報ではなく、ルーティング条件である。

| 品質状態 | 低い場合の扱い |
|----------|----------------|
| 文脈品質 | 診断・ヒアリングへ戻す |
| Story品質 | Story を再作成する |
| Architecture品質 | Architecture DAG へ戻す |
| Spec品質 | Spec DAG へ戻す |
| テスト可能性 | 実装に入らない |
| E2E体験品質 | Shipped にしない。UI操作証跡を作り、未達箇所を修正する |
| Visual / Polish品質 | Verified にしない。残差、表示崩れ、導線の違和感をGateに残す |
| 証跡品質 | Shipped / Verified にしない |

「品質が低いが実装で吸収する」は禁止する。

特に UI / Web app / SaaS の Story では、次を満たすまで完了扱いにしない。

- 主要ボタンが実際に押せる
- 主要導線が実際に遷移する
- 入力、保存、再表示、リロード後の状態が意図通りである
- ローディング、空状態、エラー状態でユーザーが迷わない
- PC / mobile の代表 viewport で表示が破綻しない
- E2E 証跡、スクリーンショット、ログ、必要なら Visual QA 残差が残っている
- 未達の場合は、AI が「完了」と言わず、未達 Gate と次の修正に戻す

## 5. 昇格ゲート

VibePro では、作業状態を昇格制として扱う。

```text
Signal
  -> Intake
  -> Diagnosed
  -> Story Ready
  -> Architecture Ready
  -> Spec Ready
  -> Execution Ready
  -> Shipped
  -> Verified
  -> 学習済み
```

昇格条件:

| Status | 次へ進む条件 |
|--------|--------------|
| Signal | 発火理由が明確で、対象顧客または対象プロダクトが特定されている |
| Intake | 状態クラスタが仮分類されている |
| Diagnosed | リスク、制約、対応可否、次の推奨経路が明記されている |
| Story Ready | 誰が、何を、なぜ、受け入れ条件が明確である |
| Architecture Ready | 境界、責務、SSOT、人間ゲートが明確である |
| Spec Ready | command、event、state、API/schema、テスト条件が接続されている |
| Execution Ready | TDD で着手可能で、リスクとロールバックが許容範囲内である |
| Shipped | 実装と検証結果が Spec に対応し、主要E2E導線が実操作で通っている |
| Verified | 証跡が残り、ユーザーが触って理解できる95%品質と運用 KPI に接続されている |
| 学習済み | 次の改善 Story または終了判断に接続されている |

---

## 6. 後付け禁止

VibePro は、後から分かったことを前段の判断根拠として扱わない。

禁止:

- 実装後に Story の意図を成功扱いに書き換える
- テストが通ったことを理由に、受け入れ条件を満たしたことにする
- 運用で偶然うまくいった結果を、設計判断の正当化に使う
- 人間ゲートが必要だった判断を、実装差分で既成事実化する

必要な場合は、前段の文書に戻って改訂し、再度 Gate を通す。

---

## 7. レイヤー別指標

VibePro は、すべてを売上や速度だけで評価しない。Layer ごとに異なる指標を使う。

| Layer | 主な指標 |
|-------|----------|
| 文脈 | SSOT信頼度、情報欠落、制約の明確さ |
| Story | 受け入れ条件の明確さ、対象ユーザーの具体性、発火理由の強さ |
| Architecture | 境界の明確さ、責務分離、人間ゲート定義、リスク分離 |
| Spec | テスト可能性、contract 完整性、状態遷移の明確さ |
| 実行 | テスト通過、差分の小ささ、rollback 可能性、セキュリティ検証、E2E導線通過 |
| 証跡 | 証跡の完全性、再現可能性、監査可能性、スクリーンショット / 操作ログ / Visual QA 残差 |
| Growth | 品質、コスト、信頼性、利用、変更耐性の改善 |

早い段階の Layer に、後段の指標を押し込まない。

## 8. AIが自走できる指標

VibePro の指標は、人間向けの報告値では足りない。AI が自分で判断し、後から自己採点し、次の判断へフィードバックできる必要がある。

AI が自走できる指標の条件:

1. 入力が機械的に取得できる
2. 出力が構造化されている
3. 後から正解ラベルまたは検証結果を取得できる
4. 差分を次の判断ルールへ戻せる

## 9. VibePro自身をVibeProで直す

VibePro は、自分自身の運用結果を例外扱いしない。

Codex / VibePro の実行ログから、同じ確認漏れ、同じ誤判定、同じ作業戻り、同じ品質不足が繰り返し見つかった場合、それは単なるオペレーションミスではなく、VibePro の Story / Architecture / Spec / Gate のどこかが不足しているシグナルとして扱う。

自己改善の原則:

- ログ確認は「見たつもり」を許さず、対象範囲、更新時刻、取得元、未確認範囲を構造化して残す
- 実行結果の評価は、変更内容だけでなく、価値、未達、残リスク、次の改善 Story まで接続する
- 実際に動いている VibePro CLI、checkout、origin/main、Skill 群の版を確認し、修正済みと思い込まない
- generic な診断や Story 生成に、過去プロジェクト固有の語彙や前提が漏れていないことを検証する
- PR 対象差分、dirty worktree、staged 差分、実行時生成物を混ぜず、Gate の判断対象を分離する
- UI / E2E / human review / visual QA の証跡が不足する場合は、完了ではなく未達の改善 Story として扱う

この自己改善ループは、VibePro を「AI に実装させる道具」から「AI に安心して任せられる状態を作る制御面」へ近づけるための中核機能である。

VibePro の中核判断指標:

| 指標 | 定義 | 目的 |
|------|------|------|
| 本番化ギャップ捕捉率 | 実装・運用で実際に重要だった本番化ギャップのうち、Launch診断時点で検出できていた割合 | 見落としを減らす |
| 本番化ギャップ的中率 | Launch診断で挙げた本番化ギャップのうち、実装・運用で実際に重要だった割合 | 過剰警告を減らす |
| ゲート違反流出率 | 本来 Gate で止めるべきだった変更・判断のうち、実装・リリース・運用へ流出した割合 | 危険な自動化を止める |

補助指標:

| 指標 | 対象 |
|------|------|
| ルート成果合致率 | 推奨ルートと、実際に成功したルートの一致 |
| 受け入れ条件網羅率 | 受け入れ条件が Spec / Test / Evidence に接続された割合 |
| 仕様逸脱率 | 実装が Spec から逸脱した割合 |
| 前進修正率 | rollback ではなく小さな追加修正で前進できた割合 |
| 手戻り率 | Story / Architecture / Spec へ戻った割合 |
| 検証証跡合格率 | 必須証跡が揃い、再現可能だった割合 |
| E2E体験到達率 | Storyの主要導線がUI上で実操作され、意図した結果に到達した割合 |
| 最終20%自動解消率 | 人間が手で潰していた完成度ギャップを、AIとGateの反復で解消できた割合 |
| Visual QA残差改善率 | 参照体験やデザイン意図に対して、残差が反復ごとに改善した割合 |
| 介入別KPI改善率 | 各改善介入が運用KPIをどれだけ動かしたか |

AI の自己フィードバックは、次の差分を必ず返す。

```text
検出できた本番化ギャップ
見落とした本番化ギャップ
過剰検出だった本番化ギャップ
流出したゲート違反
効果があった介入
効果が薄かった介入
次回の診断・ルーティング・ゲート更新案
```

## 9. 人間ゲート

人間ゲートは、AI の失敗を人間が後から直す場所ではない。AI に任せてよい範囲を事前に決める制御点である。

人間ゲートが必須の例:

- 本番データ移行
- 本番切替
- 顧客通知
- 契約責任が変わる判断
- セキュリティ例外
- ロールバック不能な変更
- テンプレートの境界を超える個別対応

人間ゲートが不要な例:

- 低リスクな静的サイト診断
- 既存テンプレート内のチェックリスト実行
- Spec が確定した小さな実装
- 証跡の収集とレポート生成

---

## 10. VibePro の原則

1. Frame を飛ばして Story を作らない
2. Story を飛ばして実装しない
3. 発火理由と状態を分ける
4. 状態クラスタを切ってからルートを決める
5. 品質状態を警告ではなくルーティング条件として扱う
6. テンプレートを Story より上に置かない
7. 実行層に意味判断を持たせない
8. 昇格ゲートを満たすまで次の段階へ進めない
9. 人間ゲートが必要な判断を自動化で既成事実化しない
10. 完了は証跡と学習まで含めて判定する
11. 指標は AI が自己採点できる形で定義する
12. 指標名は日本語を正本とし、略語で混同させない
