# VibePro リポジトリ内制御基盤 Frame

## 位置づけ

この文書は、graphify 連携と CLI 化に向けた Frame 候補である。`vibepro-operating-philosophy.md` に投影されている Brainbase Graph SSOT を置き換えるものではない。採用する場合は、永続化すべき概念を Graph SSOT に昇格する。

この Frame から派生する Story の正本は NocoDB に置く。ローカル docs に Story 投影を書くことはあっても、Story SSOT は `docs/stories/` ではなく NocoDB とする。

VibePro は、コピーされた `target/` フォルダを一時的に検査する外部チェックリストではなく、対象プロダクトのリポジトリに再利用可能な証跡、ゲート、学習材料を残すリポジトリ内制御基盤として振る舞うべきである。

## Frame

VibePro は、実在するプロダクトを「AI で作った状態」から「本番で運用できる状態」へ進めるために存在する。作業単位は一回限りの診断実行ではない。作業単位はリポジトリと、その本番化準備度の履歴である。

対象リポジトリには、AI agent、人間、Brainbase が毎回同じ証跡を見られる安定した置き場が必要になる。その置き場を VibePro が管理するプロジェクト作業領域として扱う。

```text
プロダクトリポジトリ
  -> VibePro プロジェクト作業領域
  -> 文脈グラフ
  -> 本番化診断
  -> ゲートと証跡
  -> Brainbase 集約
  -> 学習と次の Story
```

## なぜ重要か

現行の `target/` と `results/` の形は PoC には有効だが、診断がプロダクト本体から切り離されて見える。次の agent が見たときに、どのソース版を診断したのか、どの証跡を使ったのか、どの結果が最新なのかを推測し直す必要がある。

リポジトリ内作業領域にすると、VibePro は本番化プロセスの一部として扱える。

- ソース版と診断結果を紐付けられる
- 文脈グラフと関連資料を AI assistant が再利用できる
- 後続実行と前回実行の差分を比較できる
- Brainbase が任意のメモを読むのではなく、複数リポジトリの状態を集約できる
- ゲート判断を、実装後・運用後に監査できる

## 運用原則

1. VibePro は永続化する成果物を、自分が管理する作業領域内にだけ書く。
2. プロダクトのソースは顧客リポジトリが正本であり、VibePro の成果物は投影と証跡レイヤーである。
3. 生の証跡と機密性の高い検査結果はデフォルトで保護する。
4. 人間向けレポートは表示用の投影であり、構造化された管理目録が機械可読な永続連携口である。
5. graphify は文脈抽出器であり、VibePro の中核ではない。
6. VibePro は本番化準備度の解釈、ゲート、証跡、次アクションへの振り分けを持つ。
7. Brainbase は VibePro の管理目録を複数リポジトリから読む。Brainbase の管理画面のために、リポジトリ探索を毎回やり直さない。

## 境界

VibePro が持つもの:

- リポジトリ内 VibePro プロジェクト初期化
- 文脈グラフと関連資料の取り込み
- 本番化診断
- リスク台帳と準備度スコア
- ゲート結果と証跡の管理目録
- Brainbase への連携口

VibePro がデフォルトでは持たないもの:

- 顧客リポジトリのプロダクトコードを書き換えること
- 明示的な導入操作なしに assistant instructions や hooks を変更すること
- 生のセキュリティ証跡をコミット対象にすること
- graphify の推論関係を検証済み事実として扱うこと
- データ移行、本番切替、顧客通知、不可逆変更の人間ゲートを迂回すること

## 判断テスト

- 新しい agent は、まず VibePro 成果物を読むだけでリポジトリの本番化状態を理解できるか。
- Brainbase は、任意のレポートを読まずに最新の VibePro 実行とゲート状態を特定できるか。
- 機密性の高い証跡はデフォルトで git から外れるか。
- graphify を外す、または置き換えても VibePro の領域モデルは壊れないか。
- 文脈品質を単なる警告ではなく、次の経路を決める条件として扱えているか。
