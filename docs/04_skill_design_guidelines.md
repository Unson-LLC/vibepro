# 診断スキル設計指針

本ドキュメントは、VibePro診断スキルを作成・改善する際の設計原則をまとめたものである。

## 1. コンテキストエンジニアリング

### 原則
AIはコンテキストが限られているため、**どれだけ圧縮して一度に入力できるか**が品質を左右する。

### アンチパターン
- AIがすでに学習している内容を詳細に説明する
- 人間向けドキュメントのような冗長な記述
- 技術用語の定義や背景説明

### 推奨パターン
```markdown
# 悪い例（冗長）
## XSS対策
クロスサイトスクリプティング（XSS）とは、悪意のあるスクリプトを
Webページに注入する攻撃手法です。ユーザー入力を適切にサニタイズ
せずにHTML出力すると、攻撃者がJavaScriptコードを実行できます...

# 良い例（圧縮）
## XSS対策
- 項目: ユーザー入力のサニタイズ
- 判断基準: dangerouslySetInnerHTML未使用、またはDOMPurify適用済み
```

## 2. チェック項目の3要素構造

すべてのチェック項目は以下の3要素のみで構成する。

| 要素 | 説明 | 例 |
|------|------|-----|
| **観点** | チェックのカテゴリ | セキュリティ、パフォーマンス、品質 |
| **項目** | 具体的なチェック対象 | 環境変数管理、SQLインジェクション |
| **判断基準** | Pass/Failの明確な条件 | `.env`に秘密情報なし、かつ`.gitignore`に`.env.local`記載あり |

### 判断基準の定義ルール
- 曖昧な表現を避け、機械的に判定できる条件を記述する
- 「適切に」「正しく」などの主観的表現は禁止
- 具体的なファイルパス、設定値、パターンを明示する

```markdown
# 悪い例
- 判断基準: 秘密情報が適切に管理されていること

# 良い例
- 判断基準:
  - APIキー・パスワードは環境変数経由でのみ参照
  - `.env`ファイルはリポジトリに含まれない
  - 本番環境変数はVercel/Cloudflareの環境変数機能で管理
```

## 3. 点数制評価

### 評価軸
各観点について100点満点で評価し、総合スコアを算出する。

| 観点 | 重み | 説明 |
|------|------|------|
| セキュリティ | 40% | 脆弱性、認証、データ保護 |
| 品質 | 30% | コード品質、型安全性、テスト |
| パフォーマンス | 20% | 最適化、バンドルサイズ |
| 運用性 | 10% | デプロイ、監視、ログ |

### 出力ルール
- 全項目のスコアを一覧表示
- **70点未満の項目のみ**詳細な理由と改善案を記述
- 70点以上の項目は一行サマリーのみ

## 4. 階層化アーキテクチャ

診断スキルは以下の階層構造で設計し、再利用性を高める。

```
レイヤー1: アーキテクチャ
├── Web
├── Mobile
└── Desktop

レイヤー2: 実行形態
├── 静的（Static）
└── 動的（Dynamic）

レイヤー3: データ構成
├── DBなし
├── BaaS（Firebase, Supabase）
└── 自前DB（PostgreSQL, MySQL）

レイヤー4: フレームワーク
├── Next.js
├── Nuxt
├── Remix
└── Astro

レイヤー5: 言語（省略可）
├── TypeScript
├── JavaScript
└── その他
```

### 適用ルール
1. 最初のアーキ分析で全レイヤーをマッピング
2. 該当するレイヤーのスキルを組み合わせて適用
3. 対応スキルがない場合はAIの汎用知識で対応

## 5. スキルファイル構成

```
.claude/skills/{skill-name}/
├── SKILL.md           # スキル定義（3要素構造のチェックリスト）
└── references/        # 補足情報（必要最小限）
    ├── checklist.md   # チェック項目一覧
    └── *.md           # 個別の判断基準詳細
```

### SKILL.mdの記述ルール
- 冒頭に目的と対象を1-2行で記述
- チェック項目は表形式で一覧化
- 詳細説明は`references/`に分離し、必要時のみ参照

## 6. 改善プロセス

新しいスキルを作成・改善する際は以下の手順を推奨する。

1. **ドラフト作成**: 会議録やトランスクリプトをClaudeに投げて方針を相談
2. **圧縮**: 3要素構造に沿って冗長性を排除
3. **検証**: 実際のコードベースで診断を実行し、結果を確認
4. **共通化**: 他スキルと重複する部分を抽出して共通パーツ化
