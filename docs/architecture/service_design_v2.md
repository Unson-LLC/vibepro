## 1. VibePro 再定義

### 旧定義

> Vibe Coding で作られたコードの商用化を支援するサービス

### 新定義

> VibePro = AI プロダクトを最短で立ち上げ、本番で止まらず運用できる状態まで持っていく支援サービス

表のブランドメッセージは「AI 新規開発・商用化支援」に絞る。いきなり「新規開発から運用まで全部やる」と言うと焦点がぼけるため、外向きには「立ち上げ」と「本番化」を中心価値として打ち出す。

ただし、実際の提供範囲は 3 フェーズ持つ。

1. **新規開発**: AI を前提に、最短で形にする  
2. **商用化**: 本番で止まらない品質・運用に整える  
3. **運用改善**: リリース後の改善を継続できる状態にする

外向きの一言は狭く、提供範囲は広く持つ。これによりブランドの焦点を保ちながら、案件の実態に合わせた提案ができる。

### 解決する問題

対象顧客が抱える課題:

1. AI を使った新規開発をしたいが、要件整理と初期設計が弱い
2. 試作や社内版まではできたが、本番品質・セキュリティ・運用が不十分
3. モデル更新や API 変更に追従できない
4. 利用ログを改善に活かせない
5. コスト最適化ができない
6. 精度が徐々に劣化する

核心は 2 つある。

- AI プロダクトは「作る」だけでは事業にならない
- AI プロダクトは放置すると劣化する

したがって VibePro は、開発・商用化・運用の分断を埋める存在として定義する。

## 2. SSW（Super Specific Who）

### ターゲット 4 象限

```
                  技術力 高
            ─────────────────
            │ ①大手Tech企業  │ ②スタートアップ(強)
AI依存度 高 │ （内製可能）   │ （爆速開発型）
            │ →除外         │ →一部対象
            ─────────────────
            │ ③業務DX企業    │ ④AI導入中堅企業
AI依存度 低 │ （PoC止まり） │ （最有力）
            │ →教育向け     │ →最有力
            ─────────────────
                  技術力 低
```

本命は④だが、入口は「AI を使った新規開発をしたいが、本番化まで見据えてパートナーを探している会社」も含める。

### SSW 定義

**一文**: 「AI プロダクトを立ち上げたい、または立ち上げたが本番運用で詰まり始めた責任者」

### SSW の成立条件

- AI を事業で使う意思決定が済んでいる
- 新規開発中、もしくは beta 以上のプロダクトがある
- 本番化・品質・運用のどこかで不安を抱えている
- 内製だけで回し切るのが難しい
- 専任 MLOps / SRE / Platform チームを持てる規模ではない

### 3 つのペルソナ

#### SSW-A: AI 新規開発の事業責任者 / PM / PO

- AI を使った新規サービスや社内プロダクトを立ち上げたい
- 受託丸投げではなく、AI 前提で短く作りたい
- ただし本番設計や運用設計までは自信がない
- 買う理由: 最短で形にしたいが、後で作り直し地獄には入りたくない

#### SSW-B: AI プロダクト責任者（PM / PO / 事業責任者）

- 生成 AI 機能を本番提供済み or beta 以上で外部提供済み
- 精度 / コスト / 障害のどれかでクレーム予兆が出ている
- 「改善ロードマップ」が回せていない
- 困っているのは「開発」ではなく「運用の継続性」
- 買う理由: 炎上前に止血したい。数字で改善を示したい

#### SSW-C: 情シス・セキュリティ責任者（CISO / 情シス Mgr）

- 生成 AI を社外向けに出したことで監査・リスクが急増
- ベンダー / 内製が場当たりで、証跡・SLA・監視が弱い
- 「責任の所在」が曖昧で怖い
- 買う理由: 事故を起こしたくない。監査で落ちたくない

#### SSW-D: 受託開発会社の事業責任者（AI 受託 / SI の部長）

- クライアントから「出した後も見て」と言われ続けている
- 案件が単発で粗利が積み上がらない
- 運用改善を自社で抱えると死ぬ
- 買う理由: 月額化して利益を積みたい。運用を外部化したい

### 1 点突破 SSW（最初に狙う）

> AI を使った B2B プロダクトを立ち上げたい、または立ち上げたが本番化 / 運用で詰まり始めた責任者（従業員 20〜300）

### SSW 判定シグナル

強いシグナル（優先）:

- 「新規サービス」「新規事業」「立ち上げ」「MVP」「ベータ版」
- 「生成 AI」「LLM」「RAG」「AI エージェント」「問い合わせ自動化」「要約」「文書検索」
- 「SaaS」「プロダクト」「提供開始」「正式リリース」
- 「SOC2」「ISMS」「監査」「セキュリティ強化」
- 「MLOps」「監視」「SRE」「SLA」「コスト最適化」

弱いシグナル（除外寄り）:

- 「PoC」「実証実験」「検討」「勉強会」「セミナー」「研修」

### 除外すべきターゲット

- ChatGPT を業務で使っているだけの企業
- AI に興味があるだけの企業
- まだ PoC 段階で事業化意思が弱い
- AI 内製チームが 10 人以上いる会社

### SSW の生態と心理

日々の行動:

- 朝はエンジニアとの進捗確認
- 昼には経営陣から事業成長の追求
- 夜には障害や精度低下への不安

最大の悩み:

- 立ち上げが遅い、または立ち上げた後に品質 / コスト / 障害 / 運用で詰まる → 競争力を失う

刺さるメッセージ:

> 「AI で作るだけで終わらせず、最短で立ち上げて、本番で止まらず運用できる状態まで持っていく」

## 4. 価格設計

### 戦略的決定事項

- 入口（診断）は速く・軽く
- 実装は型で粗利確保
- 利益は月額で積む
- 最低契約期間: 基本 12 ヶ月、例外 6 ヶ月（割高）
- 診断費は実装に 100% 充当（有効期限 14 日）
- 実装のみ希望の場合は実装単価を 2 倍（単発化防止）
- 6 ヶ月契約は +25% の割高設定

### Launch 診断（入口）

| プラン | 価格 | 期間 |
|--------|------|------|
| ライト | ¥50万 | 1週間（対象を絞る） |
| 標準 | ¥80万 | 2週間 |
| 拡張 | ¥150万 | 3週間（複数部署調整はやらない） |

固定成果物:

- Production Readiness Score（0-100）
- Top10 リスクと対策
- 30 日実装計画（S/M/L のどれが適切か結論）
- 月額運用で追う KPI 案（精度 / コスト / 障害 / 利用）

充当ルール: 100% 充当、期限 14 日。期限超えは充当失効。

### Launch 実装（Replatform 型パッケージ）

既存に手を入れるのではなく、標準基盤に載せ替える。

| プラン | 価格 | 期間 | 移植パターン |
|--------|------|------|--------------|
| S | ¥300万 | 2週間 | Greenfield（新基盤に新実装） |
| M | ¥600万 | 4週間 | Strangler（段階移行） |
| L | ¥900万 | 6週間 | Hybrid（複雑統合） |

S の範囲: 最小の本番化（CI/CD・監視最小・権限/ログ最小・リリース可能状態）

M の範囲: Runbook、アラート設計、品質ゲート、コスト初期最適化、簡易負荷検証

L の範囲: 複雑統合（外部システム連携、複数環境、権限モデル複雑）。監査資料作り込みはオプション

実装のみ（運用なし）を希望する場合: 単価 2 倍。

### Growth Ops（月額運用 = 収益の柱）

| プラン | 価格（12ヶ月） | 価格（6ヶ月） | 内容 |
|--------|----------------|---------------|------|
| ライト | ¥30万/月 | ¥37.5万/月 | 月次レポート + 月1MTG + QA軽微 |
| スタンダード | ¥60万/月 | ¥75万/月 | KPIレビュー隔週 + 実験設計 + 改善 + 改修枠月12h |
| プレミアム | ¥120万/月 | ¥150万/月 | 優先対応 + 継続最適化 + 改修枠月24h + インシデント一次対応 |

支払い: 一括 or 月払い（途中解約は残額請求）。

解約すると失うもの（顧客に伝えるべき事実）:

- モデル更新遅延
- API 仕様変更対応遅延
- 精度改善停止
- コスト増大
- セキュリティリスク上昇
- 競争力低下

### 運用 KPI（月次レポート固定セット）

| カテゴリ | 指標 |
|----------|------|
| 品質 | タスク成功率 / 不正確率（評価セットで定義） |
| コスト | 1リクエストあたりコスト / 月次コスト推移 |
| 体験 | 利用率（アクティブ）/ コンバージョン |
| 信頼性 | 障害件数 / 重大インシデント / MTTR |
| 変更耐性 | モデル更新・API変更の対応リードタイム |

### オプション

| 項目 | S | M | L |
|------|---|---|---|
| SSO/SCIM | ¥150万 | ¥250万 | ¥350万 |
| 監査・脅威モデリング資料 | ¥100万 | ¥150万 | ¥200万 |
| データ分類/ポリシー整備 | ¥100万 | ¥150万 | ¥200万 |
| 大規模リファクタ/再設計 | 別途見積 | - | - |

## 5. アーキテクチャ方針

### 基本方針: Cloudflare-first + Escape Hatch（AWS）

標準テンプレは Cloudflare（フロント、LLM ゲート、ログ/評価、運用ループ）。やり切れない要件だけ AWS に逃がす。

「Cloudflare を OS にして、AWS を臓器として使う」

### Cloudflare で勝てる領域（VibePro 向き）

- LLM API の前段: ルーティング、フェイルオーバー、コスト制御、レート制限
- プロンプト / 評価 / ログ収集・マスキング
- Edge での軽い認証・WAF・Bot 対策
- RAG の軽〜中規模構成（R2 + Vector/検索 + Workers）
- 小〜中規模の B2B SaaS 機能としての生成 AI

### AWS 併用が必要な場合（判定チェック）

YES が 2 つ以上なら AWS 併用前提:

- 主要データが閉域 / オンプレにある
- RDB が中心で複雑クエリが多い
- 重いバッチ / 長時間処理が必須
- GPU 推論を自前で持つ
- 監査要件が強く「説明可能性」が最重要

### テンプレ 3 層構造

#### Layer A: Platform Template（共通基盤 — 一切いじらない）

- IaC（Terraform 固定）
- ネットワーク標準（Cloudflare WAF、TLS、DNS）
- Identity 標準（最小権限、Secrets 管理、鍵ローテ）
- Observability 標準（ログ・メトリクス・トレース・アラート）
- CI/CD 標準（ビルド・テスト・デプロイ・ロールバック）
- セキュリティ標準（SAST/DAST、脆弱性スキャン、SBOM）

#### Layer B: LLMOps Template（差別化の中心）

- プロンプト管理（バージョニング、差分、ロールバック）
- 評価基盤（golden set、回帰テスト、品質ゲート）
- ルーティング（モデル切替、フェイルオーバー）
- コストガード（予算、レート制御、キャッシュ）
- ログ設計（PII、マスキング、監査ログ）
- RAG 標準（埋め込み更新、再インデックス、評価）

#### Layer C: App Adapter（個別の薄い層）

- API / DB 接続
- UI / 業務フロー
- ドメインルール

顧客固有はここだけ。ここが厚くなる案件は除外。

### 固定納品物（契約別紙）

- IaC リポジトリ（環境再現 100%）
- CI/CD パイプライン（dev/stg/prod）
- 監視ダッシュボード + アラート
- Runbook（障害・ロールバック・鍵/秘密管理）
- LLM 評価セット + 回帰テスト
- コストダッシュボード（1 リクエスト単価まで）
- セキュリティチェック（最低限の項目リスト）

## 6. 移行設計

### 移行の位置づけ

Launch 実装は Replatform（標準基盤への載せ替え）。既存コードの改修ではない。

移行プロセス: GitHub 読み込み元（ユーザーの Vibe Coding コード）→ GitHub 書き込み先（VibePro テンプレート）。テンプレートの GitHub は CI/CD + Cloudflare デプロイ済み。

ただし、ここでいう「移行」は一律の標準処理ではない。

- 移行可能性の診断
- 移行方式の設計
- 低リスクな移植

は VibePro の標準フローに含められる。

一方で、次は常に Human Gate を前提とする。

- データ移行
- 本番切替
- 顧客通知
- ロールバック責任

したがって、VibePro が移行を商品として扱うことと、Auto Mode が移行を標準処理できることは同義ではない。

### 移行パターン分類

| パターン | 対応プラン | 内容 |
|----------|-----------|------|
| Greenfield | S | 既存を触らず新環境に同等機能で再構築 |
| Strangler | M | ゲートウェイを噛ませて機能単位で段階移行。条件付きで受ける |
| Hybrid | L | 複雑統合。テンプレを崩さない範囲だけ条件付きで受ける |

補足:

- `Greenfield` は標準化しやすい中心パターン
- `Strangler` は設計対象としては標準に含められるが、実行は条件付き
- `Hybrid` は商品メニューには置けても、常に標準フローに入るわけではない

### 受注条件（型化を守るための制約）

受ける条件:

- 重要機能が最大 3 フロー以内
- 本番ユーザーが〜5,000 程度
- データ移行が軽い
- 情シス監査資料の作り込みはオプション

受けない条件:

- 既存コードを尊重して「改修」が主体
- 依存システムが 5 以上
- データ移行が重い（大量・整合性・バッチ多数）
- 24/365 のオンコール要求

### 移行リスクへの対処

- 移行前にバックアップを必ず取得
- テスト環境で移行リハーサル実施
- 切り戻し手順の事前整備
- 責任範囲は契約で明確化
- 重大データは段階的に移行
- 並行運用期間を設ける

Guardrail 観点:

- `requires_data_migration`
- `has_rehearsal_plan`
- `has_rollback_plan`
- `has_parallel_run_option`
- `has_customer_communication_plan`
- `blast_radius_of_cutover`
- `contractual_liability_exceeds_template_scope`

これらを満たさない移行は、通常の実装フローとして扱わない。

### 未解決課題

- データ移行失敗時の責任範囲の定義
- 移行込みで現行価格がペイするかのシミュレーション
- SalesTailor を自社テスト案件として Cloudflare 移行を検証する案

## 7. スキル設計（診断 + 実装）

### 診断スキル（現行 + 拡張）

現行（静的サイト向け）:

- /static-site-check → セキュリティ・構成チェック
- /cloudflare-pages-deploy → デプロイ計画
- /risk-register → リスク台帳
- /estimate → 見積もり
- /diagnose → 上記一括実行

拡張方向:

- LLMOps 診断（プロンプト管理、評価基盤、コスト管理の有無）
- API 依存診断（モデル切替、フェイルオーバーの有無）
- 運用成熟度スコアリング

### 実装スキル（新規設計）

診断で炙り出された「何ができていないか」を、テンプレ上に自動構築するスキル群。

- /scaffold-platform → Layer A（Platform Template）の展開
- /scaffold-llmops → Layer B（LLMOps Template）の展開
- /migrate-app → Layer C（App Adapter）への移植支援
- /verify-migration → 移行後の検証（テスト実行、差分確認）

### チェックリスト駆動

診断時: チェックリストで商用化に何が足りないかを炙り出す。スキルの判断でチェックを確実にこなし、レポートを生成。

実装時: 同じチェックリストを使い、テンプレ上で全項目が充足されたことを検証。

## 8. サービスフロー（リード獲得 → 運用）

### Phase 0: リード獲得

- SalesTailor で SSW シグナルに基づくターゲット抽出
- フォーム送信 → 初回接触
- フィルタ: 業種（B2B SaaS / DX 受託 / AI 受託）、規模（20〜300）、キーワード

### Phase 1: 営業 → 契約

1. 初回ヒアリング（無料・30 分）→ 規模判定
2. Launch 診断の提案（¥50〜150 万）
3. 診断実行 → Production Readiness Score + Top10 リスク + 実装計画
4. Go/NoGo 判断（診断費は実装に 100% 充当、14 日期限）

### Phase 2: Launch 実装（Replatform）

1. S/M/L 確定 → 契約（実装 + Growth Ops 12 ヶ月セット）
2. GitHub 読み込み元（ユーザーコード）→ 書き込み先（テンプレート）
3. テンプレ展開 → 移植 → 検証 → Cloudflare デプロイ
4. 納品物チェックリストで完了確認

注記:

- 上記は low-risk 領域を前提にした標準フロー
- データ移行、本番切替、顧客通知を伴う高リスク移行は別スコープ審査と Human Gate を通す
- 商品として Replatform を扱っていても、Auto Mode が全移行工程を直接処理するわけではない

### Phase 3: Growth Ops（月額運用）

1. 運用開始日を実装契約時点で確定
2. 月次 KPI レポート（品質 / コスト / 体験 / 信頼性 / 変更耐性）
3. 改善サイクル（チケット + 実験駆動、会議は隔週で十分）
4. 改修枠は時間上限で契約に明記

## 9. 省エネ経営との整合

### 目指す構造

高単価 × 月額 × 継続 × 高粗利

### 条件

- 運用を自動化・テンプレ化できるか → スキル + DAG で型化
- 改善フレームを標準化できるか → KPI 固定セット + チケット駆動
- 人依存を減らせるか → テンプレ 3 層 + スキル自動実行
- 契約を継続前提で設計できるか → 12 ヶ月基本 + 実装とセット

### リスク

1. 商用化だけで終わる設計にすると単発化する → 実装 + 運用セット契約で防止
2. 運用が「保守作業」になると内製される → Growth Ops として成長価値を出す
3. 改善価値を定量化できないと継続されない → KPI 月次レポート
4. 人依存になると省エネにならない → テンプレ + スキル + 時間上限

---

## 10. アーキテクチャ原則（04 から再掲 + 追加）

1. Frame を飛ばして Story を作らない
2. Story を飛ばして Event を実行しない
3. テンプレートを Story より上に置かない
4. 実行層に意味判断を持たせない
5. reject も監査対象にする
6. DAG が積み上げるのは知能ではなく制御能力
7. Auto Mode の成熟度は許容範囲の拡張で測る
