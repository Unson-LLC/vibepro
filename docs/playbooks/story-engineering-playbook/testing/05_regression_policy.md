# 回帰テスト方針

## 1. 回帰テストの目的

回帰テストは、修正によって既存機能が壊れていないこと、過去のバグが再発していないことを確認するために行います。

## 2. 回帰テストを追加するタイミング

- バグが発生したとき
- 仕様変更で既存機能に影響があるとき
- DB/API/権限に変更があるとき
- 共通コンポーネントを変更したとき
- 状態管理を変更したとき
- 大きめのリファクタを行ったとき
- リリース後に問い合わせが発生したとき

## 3. バグ修正時の流れ

```text
1. バグ再現
2. 再現条件を記録
3. 可能なら自動テストを追加
4. 自動化しにくい場合は回帰テスト候補に追加
5. 修正
6. CI確認
7. PRレビュー
8. ステージング確認
```

## 4. 回帰テスト候補テンプレート

```md
# 回帰テスト候補

| ID | 発生日 | 機能 | バグ内容 | 再現条件 | 再発防止方法 | 自動/手動 | 対応状況 |
|---|---|---|---|---|---|---|---|
| R-001 |  |  |  |  |  |  | 未対応 |
```

## 5. 自動化する基準

以下はできるだけ自動化します。

- 同じバグが再発しやすい
- 業務影響が大きい
- APIやDBで再現できる
- 権限・認証に関わる
- 金額・日付・状態遷移に関わる

## 6. 手動確認で残してよいもの

- 見た目の違和感
- 文言のニュアンス
- スマホでの操作感
- 顧客や現場の受け止め方
