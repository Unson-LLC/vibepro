# テスト種別

## 1. 単体テスト

### 目的

関数やserviceなど、単体のロジックが仕様通りに動くことを確認します。

### 対象

- 関数
- service
- validation
- 権限判定
- 日付計算
- 金額計算
- ステータス遷移
- フォーマット変換
- 純粋なビジネスロジック

### 書くべきケース

- 正常系
- 異常系
- 境界値
- 条件分岐
- 仕様変更で壊れやすい処理

### 向いていないもの

- 実際のDB接続
- API全体の挙動
- 画面操作
- 複数機能をまたぐ導線

---

## 2. 統合テスト

### 目的

API、DB、認証、権限など、複数の処理がつながった状態で仕様通りに動くことを確認します。

### 対象

- API route
- 認証
- 権限
- DB接続
- バリデーション
- レスポンス形式
- DB更新後の状態

### 書くべきケース

- 200 / 201 の正常系
- 400 不正入力
- 401 未ログイン
- 403 権限なし
- 404 対象データなし
- 409 重複・競合
- DB更新の結果
- レスポンス形式

### 向いていないもの

- UI表示
- 細かい文言
- 画面遷移全体
- 見た目の確認

---

## 3. E2Eテスト

### 目的

ユーザーの操作に近い形で、主要導線が壊れていないことを確認します。

### 対象

- ログイン
- 主要な登録・更新・削除導線
- 決済
- 管理画面の重要操作
- 現場で必ず使う導線
- 複数画面・複数APIをまたぐ操作

### 書くべきケース

- 最重要導線の正常系
- 権限による画面表示差分
- ユーザーが実際に行う一連の操作

### 向いていないもの

- API異常系の網羅
- 単一関数のロジック
- 低頻度の細かい操作
- レイアウトの細かい差分

---

## 4. 手動確認

### 目的

自動テストでは確認しにくい、見た目・文言・操作感・現場での使いやすさを確認します。

### 対象

- UI
- 文言
- スマホ表示
- 画面の違和感
- ローディング表示
- エラー表示
- 操作感
- 現場で使いやすいか

---

## 5. 回帰テスト

### 目的

修正や追加によって、既存機能が壊れていないこと、過去のバグが再発していないことを確認します。

### 対象

- 過去にバグが出た箇所
- 仕様変更で壊れやすい箇所
- 重要な業務導線
- 共通コンポーネント変更の影響範囲
- DB / API / 権限変更の影響範囲

---

## 6. 判断表

| 確認したいこと | 推奨テスト |
|---|---|
| 関数単体のロジック | 単体 |
| バリデーション | 単体 / 統合 |
| 認証・権限 | 統合 / E2E |
| DB更新 | 統合 |
| APIレスポンス | 統合 |
| 画面遷移 | E2E |
| 主要業務フロー | E2E |
| 文言・見た目 | 手動 |
| スマホ操作感 | 手動 |
| 過去バグの再発防止 | 回帰 |
