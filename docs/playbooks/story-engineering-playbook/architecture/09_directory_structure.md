# ディレクトリ構成

## 1. このドキュメントの目的

このドキュメントでは、プロジェクト全体のディレクトリ設計方針を定義します。

具体的なディレクトリ構成は、採用するアーキテクチャ、フレームワーク、プロダクト規模、既存コードの構成によって変わります。

そのため、本ドキュメントに記載する構成例はあくまで参考例であり、必ずしもすべてのプロジェクトでそのまま採用するものではありません。

重要なのは、ディレクトリ名そのものではなく、責務が混ざらないように分離することです。

---

## 2. 基本方針

- 責務ごとにディレクトリを分ける
- UI、API、業務ロジック、DBアクセス、外部連携、型定義、テストを混ぜない
- 既存構成に合わせる
- 新しいディレクトリを作る場合は、既存の責務分離と矛盾しないか確認する
- AIに実装させる場合も、この構成方針を守る
- アーキテクチャによって構成が変わる場合は、Tech Lead / CTO が判断する
- featureごとの具体的な追加・変更ファイルは `docs/features/[feature-name]/04_technical_delta.md` で定義する
- AI実装時に変更してよい範囲は `docs/features/[feature-name]/06_tasks.md` で定義する

---

## 3. 分離すべき責務

以下の責務は、可能な限り分けて管理します。

| 責務 | 内容 |
|---|---|
| 画面・ルーティング | page、layout、route、画面単位のエントリーポイント |
| UIコンポーネント | ボタン、フォーム、一覧、モーダルなどの表示部品 |
| 機能固有のUI | 特定featureでのみ使うUI、hooks、client-side logic |
| API | HTTP endpoint、request / response の入口 |
| 業務ロジック | use case、service、ドメインルール、状態遷移 |
| DBアクセス | repository、query、永続化処理 |
| 外部連携 | 外部API、SDK、Webhook、メール、決済、SaaS連携 |
| 型定義 | request、response、domain model、DTOなど |
| バリデーション | 入力検証、schema、form validation |
| エラー処理 | error class、error mapping、表示用エラー変換 |
| ログ・監視 | logger、event tracking、監視用イベント |
| テスト | unit、integration、E2E、fixtures、mocks |
| 設定 | 環境変数定義、アプリケーション設定、ビルド設定 |
| スクリプト | 開発補助、データ移行、バッチ、運用スクリプト |

## 4. 配置判断のルール

### 4.1 画面・ルーティング

- 画面単位のエントリーポイントは、採用しているフレームワークのルーティング規約に合わせる
- 画面ファイルには、画面構成とデータ取得の入口を中心に書く
- 複雑なUIや業務ロジックを画面ファイルに詰め込まない

### 4.2 UI

- 複数機能で使うUIは共通コンポーネントとして配置する
- 特定featureでしか使わないUIは、そのfeature配下に配置する
- 画像アーティファクト由来の文言、配置、class、style、scriptをそのまま流用しない
- UIライブラリや既存コンポーネントがある場合は、それに合わせる
- 共通コンポーネントを変更する場合は、影響範囲を確認する

### 4.3 API

- APIの入口と業務ロジックを混ぜない
- API route / controller には、request parsing、認証確認、service呼び出し、response整形を中心に書く
- 業務ルールはservice、use case、domain logicなどに切り出す
- 既存APIレスポンス形式を勝手に変更しない

### 4.4 業務ロジック

- UIやAPI routeに業務ロジックを書きすぎない
- 状態遷移、権限判定、金額計算、日付計算などはテストしやすい場所に置く
- DBアクセス処理と業務判断を混ぜない
- 外部API呼び出しと業務判断を密結合にしない

### 4.5 DBアクセス

- DBアクセスはrepository、query、data access layerなどの専用層に閉じる
- UIやAPI routeからDBを直接触らない
- 既存のDBアクセス方針がある場合は、それに合わせる
- DBスキーマ変更やmigration変更は、人間の承認なしで行わない

### 4.6 外部連携

- 外部APIやSDK呼び出しは、専用のintegration層にまとめる
- UIやAPI routeから外部SDKを直接呼ばない
- 環境変数、API key、secretをコードに直書きしない
- 外部連携のrequest / response変換はテストしやすい場所に置く

### 4.7 型定義

- 機能固有の型はfeature配下に置く
- 複数機能で使う型は共通の型定義に置く
- request / response / domain model / form value / DB model を混同しない
- 型の置き場所に迷う場合は、利用範囲を基準に判断する

### 4.8 バリデーション

- 入力検証はUI、API、業務ロジックのどこで必要かを分けて考える
- API入力のバリデーションは、API入口またはschema定義にまとめる
- form validationとAPI validationを混同しない
- 境界値や異常系をテストしやすい構成にする

### 4.9 エラー処理

- 内部エラー、APIレスポンス用エラー、画面表示用エラーを混同しない
- エラーの変換処理は専用の場所にまとめる
- 秘密情報や個人情報をエラーメッセージやログに出さない

### 4.10 ログ・監視

- ログ出力は共通のloggerやmonitoring wrapperに寄せる
- 各featureで独自のログ形式を作らない
- 監視に必要なイベント名、ログレベル、タイミングを明確にする

### 4.11 テスト

- 重要ロジックは単体テストしやすい場所に置く
- APIは統合テストしやすい構成にする
- UIはE2Eまたは手動確認しやすい構成にする
- fixtures、mocks、test utilities の置き場所を明確にする

## 5. 構成例

以下は、Webアプリケーションでよく使う構成例です。

この構成はあくまで参考例であり、必ずこの形にする必要はありません。
実際の構成は、採用するアーキテクチャ、フレームワーク、プロダクト規模、既存コードの構成に合わせて決定します。

```txt
src/
├── app/              # 画面、ルーティング、API route
├── components/       # 複数機能で使う共通UI
├── features/         # 機能固有のUI・hooks・client-side logic
├── server/           # 業務ロジック、DBアクセス、server-side処理
├── lib/              # 共通処理、外部連携、utility
├── types/            # 共通型
└── tests/            # テスト
```

---

## 6. この構成例を採用する場合の責務

この章は、上記の構成例を採用する場合のみ使用します。
別のアーキテクチャを採用する場合は、その構成に合わせて書き換えます。

| ディレクトリ | 責務 |
|---|---|
| `app/` | 画面、ルーティング、API route |
| `components/` | 複数機能で使う共通UI |
| `features/` | 特定機能に閉じたUI、hooks、client-side logic |
| `server/` | 業務ロジック、DBアクセス、server-side処理 |
| `lib/` | 共通utility、外部API wrapper、設定 |
| `types/` | 共通型定義 |
| `tests/` | テストコード |

---

## 7. 新しいディレクトリを作る時の確認事項

新しいディレクトリを作る前に、以下を確認します。

- 既存のどの責務にも当てはまらないか
- 既存ディレクトリに置くと責務が混ざるか
- 複数機能で使うものか、特定featureだけで使うものか
- テストしやすい構成になるか
- AIが実装時に迷わない命名になっているか
- 既存の命名規則や配置ルールと矛盾しないか
- Tech Lead / CTO が確認しているか

---

## 8. 禁止事項

- 関係ないディレクトリに実装しない
- UI、API、業務ロジック、DBアクセスを同じファイルに詰め込まない
- 既存責務を無視して新しいファイルを追加しない
- 共通化しすぎない
- feature固有のものを安易に共通ディレクトリへ置かない
- 共通コンポーネントを破壊的に変更しない
- AIに任せて新しい構成を勝手に作らせない
- 既存の構成例と異なる場合に、理由を書かずに進めない

---

## 9. featureごとのディレクトリ設計

このドキュメントでは、プロジェクト全体の方針のみ定義します。

各featureで実際に追加・変更するファイルは、以下に記載します。

- `docs/features/[feature-name]/04_technical_delta.md`
- `docs/features/[feature-name]/06_tasks.md`

`04_technical_delta.md` では、設計として追加・変更するファイルを定義します。
`06_tasks.md` では、AI実装時に変更してよい範囲として明記します。
