# 規模判定・見積もり診断

`target/` ディレクトリ内のコードを分析し、商用化に必要な規模（ライト/スタンダード/エンタープライズ）を判定します。結果を `results/scale.md` に出力してください。

## 規模の定義（サービス設計書より）

### ライト（小規模）
- **対象**: 社内ツール、MVP検証用アプリ
- **ユーザー規模**: 100ユーザー未満
- **技術スタック**: シンプル（Next.js + Supabase等）
- **インフラ**: シングルリージョン、自動スケール不要
- **SLA**: なし

### スタンダード（中規模）
- **対象**: 顧客向けSaaS、B2Bサービス
- **ユーザー規模**: 100〜10,000ユーザー
- **技術スタック**: 標準（Next.js + Postgres + AWS ECS）
- **インフラ**: シングル or マルチAZ、オートスケール
- **SLA**: 99.5%

### エンタープライズ（大規模）
- **対象**: 大規模SaaS、基幹系システム
- **ユーザー規模**: 10,000ユーザー以上
- **技術スタック**: 複雑（マイクロサービス、複数DB等）
- **インフラ**: マルチAZ必須、DR対応
- **SLA**: 99.9%以上

## 判定観点

### 1. 技術的複雑度
- コードベースの規模（ファイル数、行数）
- 使用技術の数と複雑さ
- マイクロサービス vs モノリス
- データベースの数と種類

### 2. 想定ユーザー規模
- 認証・ユーザー管理の設計
- データモデルからの推測
- 同時接続数への考慮

### 3. 可用性要件
- 現在の障害対策レベル
- データの重要度
- ビジネスクリティカル度

### 4. セキュリティ要件
- 取り扱うデータの機密性
- コンプライアンス要件の有無
- 認証・認可の複雑さ

### 5. 現状と目標のギャップ
- 各診断結果からの総合評価
- 商用化に必要な作業量

以下の形式で `results/scale.md` に出力してください：

```markdown
| 可用性要件 | 低/中/高 | ... |

## 現状分析

| カテゴリ | 使用技術 | 複雑度 |
|----------|----------|--------|
| フロントエンド | Next.js | シンプル |
| バックエンド | ... | ... |
| データベース | ... | ... |
| インフラ | ... | ... |

### コードベース規模
- 総ファイル数: X files

- 主要ディレクトリ数: X

## 価格見積もり（参考）

| フェーズ | 価格 | 期間 |
|----------|------|------|
| A. 診断 | ¥XX万 | X週間 |
| B. 商用化 | ¥XX万 | X週間 |

| **初期合計** | **¥XXX万** | **X週間** |

※ 詳細見積もりは診断完了後に提示

## 商用化に必要な主要作業

1. [作業項目1]
2. [作業項目2]
3. ...

### オプション作業（追加見積もり）
1. [オプション1]
2. [オプション2]

## 規模アップの判断基準

以下の条件に該当する場合、上位規模を検討：
- [ ] ユーザー数がXXX人を超える見込み
- [ ] SLA 99.X%以上が必要
- [ ] 個人情報・決済情報を大量に扱う
- [ ] 24/365の監視体制が必要

## 次のステップ

1. 診断結果の確認・質疑
2. 規模の最終確定
3. 商用化フェーズの詳細見積もり
4. 契約・キックオフ
```

## 実行手順

1. `target/` ディレクトリの構造とコード量を分析
2. 技術スタックを特定
3. 他の診断結果（security, code-quality, architecture, operations）があれば参照
4. 規模を判定
5. `results/scale.md` に結果を出力
6. 完了メッセージを表示
