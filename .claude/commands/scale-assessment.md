# サイト規模判定

指定したコードを分析し、サービス規模を3段階で判定します。

## 規模レベル定義

| レベル | ユーザー規模 | 対象 | SLA |
|--------|-------------|------|-----|
| **ライト** | 100未満 | 社内ツール・MVP | なし |
| **スタンダード** | 100〜10,000 | B2B SaaS | 99.5% |
| **エンタープライズ** | 10,000以上 | 基幹系・大規模SaaS | 99.9%+ |

---

## 実行手順

### Step 1: 前提となる診断結果の確認

以下の診断結果ファイルを読み込む（存在しない場合は先に実行を促す）:

| ファイル | 内容 | 必須 |
|----------|------|------|
| `results/count-lines-of-code.md` | コード規模（行数・ファイル数） | 必須 |
| `results/detect-framework.md` | サイト構成・技術スタック | 必須 |

**ファイルが存在しない場合**:
```
規模判定を行うには、先に以下のコマンドを実行してください:
- /count-lines-of-code
- /detect-framework
```

### Step 2: 規模判定指標の収集

#### 2.1 コード規模

**`results/count-lines-of-code.md` から取得**:
- 総ファイル数
- 総コード行数
- 言語別内訳

| 指標 | ライト | スタンダード | エンタープライズ |
|------|--------|-------------|-----------------|
| ソースファイル数 | 〜20 | 20〜100 | 100+ |
| 総コード行数 | 〜5,000 | 5,000〜50,000 | 50,000+ |
| ディレクトリ深度 | 〜3 | 3〜5 | 5+ |

#### 2.2 サイト構成

**`results/detect-framework.md` から取得**:
- 種別（静的サイト / 動的アプリケーション）
- フレームワーク
- ビルド要否
- 技術スタック

| 種別 | 判定 |
|------|------|
| 静的サイト（ビルド不要） | ライト |
| 静的サイト（ビルド必要） | ライト〜スタンダード |
| 動的アプリケーション | スタンダード〜エンタープライズ |

#### 2.3 技術スタック複雑度（target/ を追加調査）

**A. データベース構成**

| 構成 | 判定 |
|------|------|
| なし / SQLite / ローカルストレージ | ライト |
| 単一DB（Postgres/MySQL/MongoDB） | スタンダード |
| 複数DB / Redis / 検索エンジン | エンタープライズ |

検出方法:
- `prisma/schema.prisma` - データソース数
- `docker-compose.yml` - DB サービス数
- package.json の DB 関連パッケージ

**B. アーキテクチャ**

| 構成 | 判定 |
|------|------|
| モノリス（単一アプリ） | ライト〜スタンダード |
| モジュラーモノリス | スタンダード |
| マイクロサービス | エンタープライズ |

検出方法:
- 複数の `package.json` / `go.mod` の存在
- `docker-compose.yml` のサービス数
- Kubernetes マニフェストの存在

**C. 認証・認可**

| 構成 | 判定 |
|------|------|
| なし / Basic認証 | ライト |
| OAuth / JWT / セッション | スタンダード |
| RBAC / ABAC / マルチテナント / SSO | エンタープライズ |

検出方法:
- `next-auth` / `passport` / `auth0` の使用
- `middleware.ts` の認証ロジック
- `roles` / `permissions` 関連コード

#### 2.4 インフラ構成

**A. デプロイ設定**

| 構成 | 判定 |
|------|------|
| 静的ホスティング / Vercel / Netlify | ライト |
| Docker + 単一サーバー | スタンダード |
| Kubernetes / ECS / マルチAZ | エンタープライズ |

検出ファイル:
- `vercel.json` / `netlify.toml` → ライト〜スタンダード
- `Dockerfile` → スタンダード
- `k8s/` / `kubernetes/` / `helm/` → エンタープライズ
- `terraform/` / `pulumi/` → スタンダード〜エンタープライズ

**B. CI/CD**

| 構成 | 判定 |
|------|------|
| なし / 手動デプロイ | ライト |
| GitHub Actions / GitLab CI（単純） | スタンダード |
| 複雑なパイプライン / カナリア / Blue-Green | エンタープライズ |

検出ファイル:
- `.github/workflows/` のファイル数・複雑度
- `.gitlab-ci.yml`
- `Jenkinsfile`

**C. 監視・ログ**

| 構成 | 判定 |
|------|------|
| なし / console.log | ライト |
| Sentry / LogRocket / 基本ログ | スタンダード |
| APM / 分散トレーシング / ELK / Datadog | エンタープライズ |

検出方法:
- `@sentry/nextjs` / `winston` / `pino`
- `opentelemetry` / `dd-trace`

#### 2.5 外部連携・API

| 構成 | 判定 |
|------|------|
| 外部API 0〜2個 | ライト |
| 外部API 3〜10個 | スタンダード |
| 外部API 10個以上 / 複雑な連携 | エンタープライズ |

検出方法:
- 環境変数の `*_API_KEY` / `*_SECRET` の数
- `fetch` / `axios` の外部ドメイン呼び出し

#### 2.6 テスト

| 構成 | 判定 |
|------|------|
| なし / 手動テスト | ライト |
| ユニットテスト / 基本E2E | スタンダード |
| 統合テスト / 負荷テスト / カバレッジ要件 | エンタープライズ |

検出方法:
- `__tests__/` / `*.test.ts` / `*.spec.ts` の数
- `jest.config.js` / `vitest.config.ts`
- `playwright.config.ts` / `cypress.config.ts`
- `k6` / `artillery` / `locust`

## 判定ロジック

### スコアリング方式

各カテゴリにスコアを付与（10カテゴリ × 1〜3点 = 最大30点）:

| カテゴリ | ライト (1) | スタンダード (2) | エンタープライズ (3) | 参照元 |
|----------|------------|-----------------|---------------------|--------|
| コード規模 | 〜5,000行 | 5,000〜50,000行 | 50,000行+ | count-lines-of-code.md |
| サイト構成 | 静的（ビルド不要） | 静的（ビルド必要） | 動的アプリ | detect-framework.md |
| DB構成 | なし/SQLite | 単一DB | 複数DB/Redis | target/ 調査 |
| アーキテクチャ | モノリス | モジュラー | マイクロサービス | target/ 調査 |
| 認証・認可 | なし/Basic | OAuth/JWT | RBAC/SSO | target/ 調査 |
| インフラ | 静的ホスティング | Docker | K8s/マルチAZ | target/ 調査 |
| CI/CD | なし | 基本Actions | カナリア/B-G | target/ 調査 |
| 監視 | なし | Sentry等 | APM/分散トレース | target/ 調査 |
| 外部連携 | 0〜2個 | 3〜10個 | 10個+ | target/ 調査 |
| テスト | なし | Unit/E2E | 負荷/カバレッジ | target/ 調査 |

### 総合判定

| 合計スコア | 判定 |
|-----------|------|
| 10〜15 | **ライト** |
| 16〜22 | **スタンダード** |
| 23〜30 | **エンタープライズ** |

## 規模別の特徴

### ライト（小規模）
- **対象**: 社内ツール、MVP検証用アプリ
- **ユーザー規模**: 100ユーザー未満
- **技術スタック**: シンプル（Next.js + Supabase等）
- **インフラ**: シングルリージョン、自動スケール不要
- **SLA**: なし
- **典型例**: 社内業務効率化ツール、プロトタイプ、PoC

### スタンダード（中規模）
- **対象**: 顧客向けSaaS、B2Bサービス
- **ユーザー規模**: 100〜10,000ユーザー
- **技術スタック**: 標準（Next.js + Postgres + AWS ECS）
- **インフラ**: シングル or マルチAZ、オートスケール
- **SLA**: 99.5%（月間ダウンタイム約3.6時間まで）
- **典型例**: 中小企業向けSaaS、Vertical SaaS

### エンタープライズ（大規模）
- **対象**: 大規模SaaS、基幹系システム
- **ユーザー規模**: 10,000ユーザー以上
- **技術スタック**: 複雑（マイクロサービス、複数DB等）
- **インフラ**: マルチAZ必須、DR対応
- **SLA**: 99.9%以上（月間ダウンタイム約43分まで）
- **典型例**: エンタープライズSaaS、基幹業務システム
